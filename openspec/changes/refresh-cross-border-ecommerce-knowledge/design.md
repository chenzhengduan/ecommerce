## Context

当前项目是一个以 `cross-border-ecommerce/` 为根目录的 Markdown 知识库，并通过 MkDocs Material 提供本地浏览页面。目录中既有 `01-基础概念/` 到 `08-实战避坑/` 这类基础模块，也有大量编号专题文件夹和每日学习记录。刷新流程必须适配这种“文件即知识库”的结构，而不是引入数据库、CMS 或复杂应用。

主要使用者是知识库维护者，以及后续执行 `/opsx:apply` 的 agent。每次执行时，它们需要按固定流程完成：检查现有内容、判断缺失主题、识别过时内容、写入或更新笔记，并验证文档站点仍可构建。

## Goals / Non-Goals

**Goals:**

- 为 `cross-border-ecommerce/` 下所有 Markdown 内容建立可重复执行的刷新流程。
- 保留现有文件夹名称、Markdown 文件和 MkDocs 行为，除非确实需要为了可发现性或正确性做调整。
- 通过“主题地图”与当前文件清单对比，识别缺失主题。
- 针对平台政策、税务合规、广告产品、物流限制、工具推荐等时效性内容识别过时风险。
- 对变化较快的事实，要求先查验当前来源，再更新笔记。
- 每次刷新后记录扫描了什么、补充了什么、更新了什么，以及哪些内容被延后处理。

**Non-Goals:**

- 本变更不构建完整爬虫、定时任务、数据库或图形界面。
- 本变更不要求一次性重写整个知识库。
- 本变更不要求所有笔记立刻统一模板。
- 本变更不替换 MkDocs Material，也不改变当前 Markdown 目录作为知识库的方式。

## Decisions

1. **以 Markdown 目录作为事实来源。**
   - 原因：当前项目已经直接用 Markdown 管理知识，并且 MkDocs 可以直接渲染这些文件。
   - 备选方案：迁移到数据库或 CMS。放弃原因是迁移成本高，也会削弱本地直接编辑 Markdown 的便利性。

2. **先生成清单，再做内容修改。**
   - 原因：每次刷新都必须先知道当前有什么，才能判断缺什么、哪些可能过时。清单应包含路径、标题、标题层级、修改时间、主题编号和分类。
   - 备选方案：只检查当前打开的文件。放弃原因是用户要求覆盖当前目录里的所有跨境电商知识。

3. **用主题地图定义期望覆盖范围。**
   - 原因：只有先定义目标知识面，才能判断“缺失”。主题地图应覆盖基础概念、平台运营、选品、Listing/SEO、广告、物流、供应链、支付、税务、合规、独立站、社交电商、数据分析、自动化、风险控制和区域市场等方向。
   - 备选方案：把现有文件列表视为完整知识体系。放弃原因是这样永远发现不了已有缺口。

4. **只对时效性内容做外部来源查验。**
   - 原因：稳定概念类笔记可以基于本地上下文整理；政策、费用、合规、平台规则、广告产品、税务和物流限制等内容必须查验当前来源。
   - 备选方案：每篇笔记都联网查验。放弃原因是成本高、噪声大，也会让每次刷新过慢。

5. **采用增量、可审阅的更新方式。**
   - 原因：知识库已有数百篇笔记，单次运行应优先处理价值最高的缺口和过时内容，并总结延后项，避免产生难以审阅的大规模改动。
   - 备选方案：每次自动重写所有文件。放弃原因是会制造大量无意义 diff，并可能覆盖原有有效内容。

## Risks / Trade-offs

- **风险：主题地图不完整** -> 缓解方式：把主题地图作为持续更新的活文档，发现新类别时同步补充。
- **风险：过时检测漏掉细微政策变化** -> 缓解方式：对时效性内容强制查验来源，并在无法确认时记录假设和待跟进项。
- **风险：生成内容风格不一致** -> 缓解方式：优先保留现有 Markdown 风格，只在新增或大幅更新笔记时逐步引入更统一的结构。
- **风险：`daily/` 学习记录让全量扫描噪声过大** -> 缓解方式：在清单中区分正式主题笔记和每日学习日志，优先处理正式主题笔记。
- **风险：内容更新后 MkDocs 构建失败** -> 缓解方式：每次更新后运行 `mkdocs build --strict`，在完成前修复编码、路径或 Markdown 问题。

## Migration Plan

1. 添加本变更定义的刷新流程产物。
2. 实施时先创建主题地图和知识清单，不立即修改内容。
3. 首次刷新采用保守模式，只新增或更新少量高优先级主题。
4. 通过人工检查 Markdown 和运行 `mkdocs build --strict` 验证结果。
5. 如首次刷新结果不符合预期，可回滚生成的内容改动，同时保留本次 proposal 相关产物继续调整。
