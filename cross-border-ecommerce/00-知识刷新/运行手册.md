# 跨境电商知识库刷新运行手册

> 用途：后续每次要刷新知识库时，按本文档执行。OpenSpec 的 `tasks.md` 是一次性建设清单；本文档才是长期重复执行的 runbook。

## 执行频率

- 常规刷新：每周或每次学习阶段结束后执行一次。
- 时效性刷新：平台政策、税务合规、物流费用或广告规则发生变化时立即执行。

## 固定流程

1. 生成知识清单。

   ```bash
   python3 tools/knowledge_refresh.py
   ```

2. 对照 [主题地图](主题地图.md)，找出缺失主题。

3. 标记过时风险主题。

   优先检查包含平台规则、费用、税务、合规、广告、物流、市场政策、工具推荐的笔记。

4. 选择本轮范围。

   默认每次只处理少量高价值主题：

   - 1 到 3 个缺失主题。
   - 1 到 3 个过时或不完整主题。

5. 查验来源。

   - 稳定概念可以基于现有知识库整理。
   - 政策、费用、税务、广告、物流和工具推荐必须查验当前来源。
   - 无法确认的内容不要写成确定结论，记录为待跟进。

6. 更新内容。

   - 已有主题优先在原文件更新。
   - 缺失主题按主题地图中的目标路径创建。
   - 保留准确的旧内容，只修改需要提升时效性、清晰度或覆盖度的部分。

7. 生成刷新报告。

   报告应记录：扫描数量、新增主题、更新主题、来源依据、延后项、构建结果。

8. 验证文档站点。

   ```bash
   .venv/bin/mkdocs build --strict
   ```

9. 更新并检查本地网页。

   - 如果 `mkdocs serve` 已运行，确认页面已热重载，并能看到本轮新增或更新内容。
   - 如果服务未运行，启动：

     ```bash
     .venv/bin/mkdocs serve -a 127.0.0.1:8000
     ```

   - 在刷新报告中记录网页验证结果。

## 完成标准

- 本轮刷新报告已生成。
- 所有新增和更新文件已列出。
- 时效性事实有来源依据或被标记为待跟进。
- `mkdocs build --strict` 通过。
- 本地网页已更新，或报告中记录了服务未运行和启动命令。
